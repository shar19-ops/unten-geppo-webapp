# 車両マスタのFirebase共有(クラウド同期)設計書

## 背景・目的

QRコードによる車両自動選択機能は、車両マスタ(社有車・私有車リスト)が`localStorage`に保存されていることが原因で、車両を登録した端末と異なる端末でQRコードを読み取ると「QRコードに対応する車両が見つかりませんでした」というエラーになる。これは`localStorage`がブラウザ・端末ごとに独立した保存領域であり、サーバー側で共有されていないために起こる。

共用の社有車を複数の社員のスマホでQR運用する、という当初の要件を満たすには、車両マスタを全端末で共有できるようにする必要がある。参考にしている馬券投票アプリ(`C:\Users\shar1\OneDrive\MCフォルダ\馬券投票アプリ\keiba-baren-v3.html`)がFirebase Realtime Databaseを使って同様の複数端末共有を実現しているため、同じ方式を踏襲する。

## 全体アーキテクチャ

Firebase Realtime Databaseを車両マスタの一次情報源(source of truth)とする。`localStorage`は表示用のローカルキャッシュ兼オフライン時のフォールバックという位置づけに変わる。

- **同期対象は車両マスタ(社有車・私有車リスト)のみ**。運転記録(日々の走行データ・給油記録・月報)は今まで通り`localStorage`のみで完結し、同期対象にしない。走行記録そのものを複数端末で共有する話は別の大きな論点(以前保留にした「共用社有車の複数ドライバー運用」に近い話)であり、本設計のスコープには含めない。
- アプリ起動時、Firebaseから車両マスタを取得してローカルキャッシュを上書きしてから画面を表示する。
- 車両リスト画面での追加・編集・削除は、Firebaseへ直接書き込み、成功したらローカルキャッシュも更新する。
- 通信できない場合はローカルキャッシュにフォールパックして今まで通り動作させる。

`storage.js`冒頭には元々「将来クラウドDBへ移行する際は、この中身を差し替えるだけでUI側は無改修で済む想定」という設計意図のコメントがあり、本設計はこれを実行するものである。ただし、Firebase通信は非同期のため、書き込み・起動時読み込みまわりの関数は非同期化が必要になる(詳細は「内部実装への影響」参照)。

## Firebaseプロジェクト

- 本アプリ専用に**新規Firebaseプロジェクトを作成する**(馬券投票アプリのプロジェクトとは別。データ・無料枠の制限を分離する)。
- Realtime Databaseを有効化する。
- セキュリティルールは馬券投票アプリと同じ**全開放**とする:
  ```json
  { "rules": { ".read": true, ".write": true } }
  ```
- Database URL(例: `https://<project-id>-default-rtdb.<region>.firebaseio.com`)は、プロジェクト作成後に管理者(ユーザー)から実装時に提供を受け、`FIREBASE_DB_URL`という定数としてJSコード内に埋め込む(`ADMIN_PASSWORD`と同じ、プレーンな定数埋め込み方式)。

## データ構造

`/vehicles/{車両ID}` というキー付きオブジェクト(配列ではなく、既存の`v.id`をキーとするマップ)として保存する。Firebase RTDBは配列よりこの形式の方が相性が良く、1台単位の追加・更新・削除がしやすい。

保存するフィールドは現状ローカルに保持している車両オブジェクトのフィールドをそのまま使う。フィールド単位のアクセス制限は行わない(ルール全開放のため):

- `id`(文字列。`generateId()`で生成)
- `vehicleType`(`'company'` | `'private'`)
- `plateNumber`(車両番号)
- `nickname`(車種・名称)
- `officeName`(事業所名)
- `defaultManager`(社有車のみ。既定の車両管理者名)
- `driverName`(私有車のみ。使用者名)
- `active`(真偽値。使用中フラグ)
- `createdAt` / `updatedAt`(ISO日時文字列)

## REST API呼び出し

Firebase SDKは使わず、素の`fetch()`のみを使用する(ビルド不要という既存方針・馬券投票アプリと同じ方式)。

- 取得: `GET {FIREBASE_DB_URL}/vehicles.json`
- 1台の追加・更新: `PUT {FIREBASE_DB_URL}/vehicles/{id}.json`(bodyは車両オブジェクトのJSON)
- 1台の削除: `DELETE {FIREBASE_DB_URL}/vehicles/{id}.json`
- Excel/JSONインポート確定時の一括反映: `PUT {FIREBASE_DB_URL}/vehicles.json`(競合解決後の全体マップで上書き)

## 同期タイミング

**読み込み(Firebase→ローカルキャッシュ)のタイミング:**

1. アプリ起動時に必ず1回、Firebaseから車両マスタを取得してローカルキャッシュを上書きしてから画面を表示する(QRパラメータの有無に関わらず。これにより初めて開く端末でも常に最新の一覧が使える)。
2. 「車両リスト」(管理者)画面を開くたびに再取得する(他の端末での追加・削除を見逃さないため)。
3. 通常のタブ切替(運転記録入力⇔運転月報)では再取得しない。起動時に取得したキャッシュをそのまま使う(車両追加はそう頻繁ではないため、通信コストを抑える)。

**書き込み(ローカル操作→Firebase)のタイミング:**

- 車両の追加・編集・削除は、その場でFirebaseへ書き込み、成功したらローカルキャッシュも更新する。
- Excel/JSONインポートは、競合解決後の最終リストをまとめてFirebaseへ一括反映する。

## エラー処理・オフライン時の挙動

- 起動時の取得に失敗(通信エラー・電波不良等)した場合は、エラーを表面化させずローカルキャッシュ(前回取得分)で表示を継続する。QR指定の車両がキャッシュにも無ければ、既存の「QRコードに対応する車両が見つかりませんでした。車両を選び直してください」を表示する(現状のエラーメッセージ・挙動を変更しない)。
- 「車両リスト」画面での追加・編集・削除がFirebaseへの書き込みに失敗した場合は、「保存できませんでした(通信エラー)」という趣旨のエラーメッセージを表示し、ローカルキャッシュは更新しない(実際にクラウドへ反映されていない変更が手元にだけ残る食い違いを防ぐため)。

## 内部実装への影響(技術的な注記)

現状`loadVehicles()`/`saveVehicle()`/`deleteVehicle()`(`storage.js`)は同期処理(`localStorage`のみ)だが、Firebase通信は非同期のため、以下が非同期化される:

- `storage.js`: Firebase取得・1台push・1台削除・一括pushを行う新しい非同期関数を追加する(既存の`loadVehicles()`はローカルキャッシュを読む同期関数のまま維持し、描画処理はこれを使い続ける)。
- `app.js`: 起動処理(QRパラメータ解決を含む)と`showView()`が非同期化される。
- `vehicles.js`: 車両の追加・編集・削除・Excel/JSONインポート確定・競合解決確定の各処理が非同期化される。

`trip-entry.js`・`report.js`は変更不要(すでに更新済みのローカルキャッシュを`loadVehicles()`経由で同期的に読むだけで済むため)。

## セキュリティ上の注意

Firebaseのルールを`{".read": true, ".write": true}`(全開放)にするため、Database URLを知っていれば、アプリの管理者パスワード画面を介さずに誰でも直接車両データを読み書きできる。Database URL自体はJSコード内にプレーンな定数として埋め込まれるため、開発者ツール等で見ようとすれば見える。

これは現行の管理者パスワード(`anzen_kanri`をJSコード内に平文で埋め込み、`sessionStorage`で解除状態を保持するのみ)と同じ「クライアント側のみの簡易保護、本格的なアクセス制御ではない」というセキュリティモデルの延長であり、馬券投票アプリと同じ前提・同じリスク許容度で採用する。

## スコープ外(別設計とする)

- 運転記録(走行データ・給油記録・月報)を複数端末で共有する改修(「共用社有車の複数ドライバー運用」に関わる、より大きな論点。別途保留)
- Firebase Authenticationなど、本格的な認証機構の導入
- 将来的な社内サーバへの移行(ユーザーから将来構想として言及があったが、本設計では着手しない。ただし`storage.js`の抽象化層を維持することで、将来の移行コストを抑える設計にはなっている)
