# 運転記録・給油記録のFirebase自動同期 設計書

## 背景・目的

現在、運転記録入力・給油入力で保存したデータ(月報レコード)は`localStorage`(端末ごとに独立した保存領域)にのみ保存されており、他の端末やPCと共有されない。そのため、PCで運転月報を開いても、他の社員がiPhoneで入力した記録が反映されないという課題がある。車両マスタは既にFirebase Realtime Databaseで複数端末共有する仕組みを導入済みのため、同じ方針で運転記録・給油記録もFirebaseへ自動送信し、複数端末で共有できるようにする。

## 全体アーキテクチャ

- 運転記録入力・給油入力で「保存」を押すと、まず今まで通りローカル(`localStorage`)へ確実に保存し、その直後にその日の記録だけをFirebaseへ自動送信する。送信の成否をユーザーに個別に確認させることはしない(バックグラウンドで行う)。
- 送信に失敗した場合(電波不良等)でも、ローカル保存は既に完了しているため運転記録が失われることはない。未送信の記録は「未送信キュー」として端末側に保持し、後述のタイミングで自動リトライする。
- 運転月報画面で車両・月を選んだ際、その組み合わせの月報データをFirebaseから取得してローカルの月報データにマージし、画面に反映する(車両マスタと同じ「開いた時に最新化」方式。開いている間の自動更新はしない)。
- 車両マスタと同様、Firebaseのルールは全開放(`{".read":true,".write":true}`)とし、Firebase SDKは使わず素の`fetch()`のみで読み書きする(ビルド不要という既存方針を維持)。

## データ構造(Firebase)

月報レコード1件(車両+年+月、既存の`monthlyLogKey`と同じキー)を、Firebase上では「日ごとのデータ」と「それ以外」に分けて保存する:

```
/logs/<key>/days/<日>   … meterReading, destination, driver, alcoholCheck, fuelAdded, updatedAt, updatedBy
/logs/<key>/meta        … vehicleId, privateCarLabel, year, month, checklistMid, checklistEnd, updatedAt
```

日ごとのデータを個別のパスに分けて書き込むことで、Aさんが5日分、Bさんが8日分を別々に保存しても、Firebase上ではそれぞれ別の場所に書き込まれ、互いの入力を上書きすることがない。日常点検結果や車両管理者関連の情報など、日をまたぐ情報は`meta`としてまとめて扱う。

## 複数人の入力の統合方法(マージルール)

- 運転月報を開く際、Firebaseから該当月報の`days`・`meta`を取得し、ローカルの月報データとマージしてから表示する。
- 日ごとのデータは、その日の`updatedAt`(更新日時)を比較し、新しい方を採用する(ローカルとクラウドのどちらが新しいかで自動判定)。
- `meta`(点検結果等)も同様に、全体の`updatedAt`を比較して新しい方をまとめて採用する。
- 同じ日に複数人が入力するケースは、既存の「既に入力済みです。上書きしますか?」という確認によって通常は防止されるが、万一発生した場合もこの自動マージルールで解決し、手動での競合解決画面は表示しない。
- 既存のExcel/JSON手動取込機能(明示的な競合解決パネル付き)は、この自動同期とは別の仕組みとして変更せずそのまま残す。手動でのファイル経由の月報移行手段として引き続き利用できる。

## 送信失敗時のリトライ

- 送信に失敗した記録は、「未送信キュー」としてlocalStorageに小さなリスト(どの車両・年・月・日の分が未送信か)を保持する。データ自体は既にローカルの月報データに保存済みであり、キューはあくまで「まだFirebaseに送れていない」という印である。
- リトライは常時監視するのではなく、以下のタイミングでのみ試みる:
  - アプリを開いた(再読み込みした)とき
  - 次に何か別の記録を保存したとき(そのついでに、残っている未送信分もまとめて送信を試みる)
- リトライに成功すればキューから該当項目を削除する。何度も失敗してもローカルのデータ自体は消えない。

## スコープ外

- 運転月報を開いている間のリアルタイム自動更新(定期的なポーリング等)は行わない。
- 既存のExcel/JSON手動取込・競合解決機能の変更は行わない。
- 車両マスタと異なり、運転記録・給油記録はデータ量が多くなりうるため、起動時に全車両・全月分を一括同期することはしない(閲覧・保存の都度、必要な月報データだけを同期する)。
- 「既に入力済みです。上書きしますか?」という既存の重複確認は、保存操作自体を電波の有無に関わらず即座に完了させるという要件を優先し、引き続き自機のローカルデータのみを見て判定する(保存前にクラウド側の最新状態を毎回問い合わせることはしない)。そのため、別々の端末からほぼ同時に同じ日を入力した場合、双方とも警告なく保存され、後から開いた運転月報では更新日時が新しい方の内容が採用される。データが壊れたり消えたりすることはないが、この場合に「他の人が既に入力済み」という警告は出せない。実運用上の発生頻度は低いと想定し、今回は許容する。
