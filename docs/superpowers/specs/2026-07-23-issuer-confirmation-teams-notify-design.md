# 発行者確認イベント(Teams通知連携) 設計書

## 背景・目的

運転月報の押印欄のうち「発行者」欄は、月末の日常点検が完了した後、発行者が月報の内容を確認して記入(押印)することを想定している。しかし、運転記録入力は主にスマホから、朝の通勤前など急いでいる場面で行われることが多く、月末点検の完了に気づいて運転月報(PCでの閲覧が主)を開く機会がほとんど無い。この課題を解消するため、月末点検の完了を契機に、社内で利用しているMicrosoft Teamsへ自動的に通知を送り、発行者が気づいて確認・記入できるようにする。

## 全体の流れ

1. スマホの運転記録入力画面で、既存の月末点検(日常点検)の仕組みを通じて点検を完了させる。運転者側の操作は従来通りで、追加の操作は発生しない。
2. 月末点検の保存をトリガーに、追加の操作なしで自動的にMicrosoft Teamsへ通知が送信される。通知には該当車両・年月と、その運転月報画面へ直接ジャンプするリンクを含める。
3. 発行者がTeams通知に気づき、リンクから運転月報を開くと、「月末点検が完了しました。内容をご確認のうえ発行者欄にご記入ください」という案内と「確認しました」ボタンが表示される(月末点検が完了していて、かつまだ発行者確認が済んでいない場合のみ表示)。
4. 「確認しました」ボタンを押すと、押印欄の「発行者」欄に「確認日(yy/m/d形式)」を1段目、「車両管理者の苗字」を2段目として自動的に表示する。氏名等の入力操作は不要(すべて既存データ・現在日時から自動的に決まる)。

## データモデル

月報レコードに`issuerConfirmedAt`(確認日時のISO文字列。未確認は`null`)を追加する。既存の日常点検結果(`checklistMid`・`checklistEnd`)と同じ`meta`情報の一部として扱い、`buildMetaPayload`に含めて既存のFirebase同期の仕組み(`syncLogMetaToCloud`・`syncMonthlyLogFromCloud`)にそのまま乗せる。新しい同期経路は作らない。

## Teams通知

- 月末点検(`checklistEnd`、15日点検の`checklistMid`は対象外)の保存が完了した瞬間に、追加の操作なしで自動的に送信する。
- 通知内容: 「[運転管理月報] {車両番号}（{車種名}）の{年}年{月}月分の月末点検が完了しました。内容をご確認のうえ、発行者欄への確認をお願いします。」+ 運転月報への直接リンク。
- 送信先はMicrosoft Teamsのワークフロー(Webhook)のURLで、`storage.js`に定数として埋め込む(既存の`FIREBASE_DB_URL`・管理者パスワードと同じ方式)。このリポジトリは公開設定のため、このURLも第三者から閲覧可能になる点は確認済みで、そのリスクを許容したうえで進める。
- 送信は一回きりのfire-and-forgetとし、失敗時のリトライは行わない(失敗しても運転月報画面のバナーが引き続き案内役になる)。

## 運転月報への直接リンク

新しいURLパラメータ(`?reportVehicle=<車両ID>&reportYear=<年>&reportMonth=<月>`)をアプリ起動時に読み取り、該当の車両・年月を選択した状態で運転月報タブを自動的に開く。既存のQRコードによる運転記録入力への車両自動選択(`?vehicle=`)と同じ考え方で`app.js`に追加する。既存の`?vehicle=`パラメータとは別のパラメータ名を使うため、互いに干渉しない。

## 発行者欄の表示

- 押印欄のテーブル(3列: 安全運転管理者・副安全運転管理者・発行者)は引き続き同じ`<table>`のまま維持するが、CSSを列単位に分ける。
  - 「安全運転管理者」「副安全運転管理者」の2列は、従来通り印刷・PDF出力時のみ表示される空欄の押印欄のまま(画面プレビューでは非表示)。
  - 「発行者」列のみ、画面プレビュー・印刷の両方で常に表示する。
- 発行者確認前は、発行者欄も他の2欄と同様に空欄のまま。
- 「確認しました」を押した後は、発行者欄に確認日(1段目、yy/m/d形式。例: 26/7/31)と車両管理者の苗字(2段目。車両リストに登録されている`vehicleManager`の姓の部分のみ)を表示する。
- 3列とも23mm角のセルサイズは統一したまま変更しない。

## スコープ外

- 「安全運転管理者」「副安全運転管理者」欄の同様のデジタル化(押印以外の方法)は今回の対象外。引き続き印刷後の手作業の押印を想定する。
- Teams通知の送信失敗時の自動リトライは行わない。
- 発行者確認後に確認を取り消す・修正する機能は設けない(確認は一度きりの操作とする)。
- 発行者が複数人いる場合の氏名の使い分けは考慮しない(確認操作を行った人の氏名は記録せず、車両管理者の苗字を表示するのみ)。
